# Project Reflection

## 1. The hardest bug I hit this week, and how I debugged it
The hardest bug was that the API overspend rule was never firing even when I intentionally entered extreme API spend numbers. The audit engine was correct for subscriptions but silent for API plans. My initial hypothesis was that the rule logic was too strict, so I lowered the threshold and added console logging inside `checkApiOverspend`. The logs never appeared. That pointed to the rule not being executed at all. I traced the flow from the form submission to `aggregateAudit`, then to `auditTool`, then into `generateRecommendations`. The input showed `planId: "api"`, but the pricing database did not define any API plan for those tools. The rule was effectively unreachable because the form could not select API plans in the first place.

I fixed this in three steps. First, I added explicit API plans to the pricing database for ChatGPT, Claude, Gemini, OpenAI API, and Anthropic API. Second, I updated the plan type model to support usage-based plans so the engine could treat API plans differently from per-seat subscriptions. Third, I rewired the rule to map usage-based tools to a realistic alternative subscription (ChatGPT Plus or Team, Claude Max or Team, Gemini Advanced or Ultra). Once those changes landed, I added a targeted unit test that verified a Claude API input now suggests Claude Max and yields a specific savings amount. The bug was less about math and more about data modeling, which was a useful reminder: deterministic logic only works if the data model is complete.

## 2. A decision I reversed mid-week, and what caused it
I initially planned to let an LLM generate the audit itself, with a prompt that included plan pricing and usage hints. On paper that sounded fast and “smart.” By day two, I reversed that decision. The trigger was a quick experiment where I fed the model a mock stack and asked for recommendations. It confidently hallucinated plan prices and even suggested downgrades that violated minimum seat requirements. The audit is financial advice; it cannot be probabilistic. That forced a change in architecture: a deterministic engine for all pricing decisions and a narrow AI summary layer for narrative only.

That reversal shaped the rest of the system. I invested in a strongly typed pricing database, explicit rule functions, and unit tests for every optimization path. It also changed how I approached documentation. Instead of marketing vague “AI insights,” I leaned into transparency: every number is sourced, and the engine is a simple ruleset you could reason about as a finance person. The LLM now consumes the deterministic output and turns it into a short executive summary. It is not allowed to calculate. The net result is a product that feels more trustworthy and aligns with Credex’s own ethos: AI is powerful, but it must be used in the right place.

## 3. What I would build in week 2
Week 2 would focus on shareability and scale. The first item is dynamic Open Graph images that display the savings headline for each public audit. That directly increases share rates on X, Slack, and Discord. I would implement a `/opengraph-image` route using Next.js `ImageResponse`, then cache the result at the edge for popular audit links to protect the database.

Second, I would add a lightweight benchmark mode. The deterministic engine already has enough data to compute “spend per seat,” but I would add a small anonymous aggregate table in Supabase to compute size-based benchmarks (for example, teams of 1-10 average $X per seat). This would require careful privacy handling and a threshold to avoid disclosing small sample sizes.

Third, I would add a PDF export of the audit report for internal finance reviews. This is a natural product-led acquisition hook: finance teams love an artifact they can share. I would generate PDFs server-side to avoid client-side performance issues and provide a “Copy for finance” CTA. Finally, I would add a background queue for AI summaries to smooth traffic spikes and protect the UX when the LLM API is under load. That would allow the deterministic results to appear instantly while the narrative summary loads asynchronously.

## 4. How I used AI tools (and one time AI was wrong)
I used AI tools for three categories of work: scaffolding, copy polish, and test coverage ideas. Copilot accelerated boilerplate in components and Zod schemas. ChatGPT helped rephrase landing copy and suggested a concise summary template for the audit. I also used an LLM to brainstorm edge cases for the audit engine tests, which improved the rule coverage. Importantly, I did not trust AI for pricing logic. All plan prices and rule thresholds were sourced and implemented manually in `pricing.ts` and verified in `PRICING_DATA.md`.

One concrete case where AI was wrong: it suggested that ChatGPT Team was $25/user/month “standard” without noting that this is an annual billing rate, not monthly. The official page listed $30/user/month for monthly billing. That correction mattered because the audit engine compares API spend to monthly subscription costs. A second error was the model’s claim that Claude “Max” did not exist; it treated it as a non-official tier. That pushed me to validate every plan against the vendor’s pricing page and document the verification date. The lesson was simple: AI is useful for speed, but pricing math and compliance require human verification.
